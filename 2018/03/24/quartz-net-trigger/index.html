Quartz.NET 作业调度（三）：Trigger

发表于 2018-03-24 | 分类于 Quartz.NET | 阅读次数:

在 Quartz 中，Trigger 即 Job 执行的触发器。和 JobBuilder 类似，我们可以通过 TriggerBuilder 来创建各种情况需要的触发器。Trigger 结合 Job，才能实现完美的作业调度。

TriggerBuilder

Identity

Trigger 和 Job 类似，都需要设置 name 和 group 来标识。如果不设置依然由其自动生成 name。

简单粗暴点，这样的 Trigger 其实就可以配合 Job 工作了，运行测试会发现 Job 会被马上执行一次，通过源码可以发现，默认开始时间是当前时间，循环一次，但这样的 Trigger 基本什么卵用。如：开始时间、结束时间、执行频率等都没有体现，基本是属于失控的触发器，所以接下来我们需要继续完善它。

StartAt & EndAt

通过 StartAt 和 EndAt 我们可以设置一个 Trigger 开始执行和结束时间，TriggerBuilder 同时预置了一个 StartNow() 方法。

UsingJobData

通过 TriggerBuilder 的 UsingJobData 可以为 Trigger 设置一些附属信息，在 Job 的 Execute 方法中，通过 context.Trigger.JobDataMap[“xxx”] 获取属性值，和 context.JobDetail.JobDataMap 使用方式一致。

WithSchedule

WithSimpleSchedule

通过 WithSimpleSchedule 可以设置 Job 执行的周期及执行的次数，不过循环周期基本是局限于 时、分、秒。

WithCalendarIntervalSchedule

针对 WithSimpleSchedule 做了一定的扩展，可以支持 年、月、周、日、时、分、秒及夏令时是设置。

WithDailyTimeIntervalSchedule

支持更精确的时间配置，可选每天、周末、工作日、周几，并指定的时间范围及执行时间周期，相比前面两种方式更加灵活。

WithCronSchedule

通过 Cron 表达式来设置执行时间与周期，语法简单，但却能实现非常丰富和灵活的时间配置。Cron 表达式一共7位，最后一位代表年，通常可以直接使用6位。

先瞄一眼下面表达式及含义：

下面对部分符号进行说明：

符号 说明
? 只在日期和星期字段中使用，天和星期必须有一个是?
* 每（秒、分、小时、天、月)
/ 递增：0/5。如果在分位，则代表在 0分，5分，10分，15分 的时候执行
- 范围 8-22 。如果在小时位，则代表在8点到22点之间执行
, 或 1,3,5,6。如果在天位，则代表每月1号，3号，5号，6号执行
L 只在日期和星期字段中使用，代表 “Last” 的意思。
“0 0 0 L ?” 表示每月最后一天；
“0 0 0 ? L” 表示每月最一个星期六（相当于7L）；
# 只能在星期字段中。使用 “0 0 0 ? * 6#3” 表示每月的第三个星期五
（6表示星期五，#3表示当前的第三个）

Misfire 机制

系统可能由于出现某些异常导致在预定时刻并没有触发 Job 执行，当系统恢复正常时，Quartz 会先检查当前时刻与预定时刻的差值，如果小于等于 misfireThreshold 值（默认60秒，可以自定义），则不认为发生 Misfire，并立刻启动该任务，而且之前错过的 Job 执行会全部补上。

WithSimpleSchedule 有 6 种 Misfire 机制：

  1. WithMisfireHandlingInstructionFireNow
    之前丢失的立即补执行一次，时间会从当前执行开始计算，时间周期不变
    剩余次数=预定的总次数-(丢失次数-1）
    (如：原设置是15时开始，循环周期是1h，循环10次。系统在14点挂了，当 16:30 恢复时，首先马上会执行一次，下一次执行时间变成 17:30，剩余10-(2【丢失2次】-1【补1次】)=9 )

  2. WithMisfireHandlingInstructionIgnoreMisfires
    之前丢失的次数会马上全部补上，之后执行时间点和周期不会发生变化
    剩余次数=预定的总次数-丢失次数
    (如：原来设置16时开始，循环周期是1h。当恢复时间是 16:30，首先马上会执行丢失的所有次数，下一次执行时间依然是 17:00)

  3. WithMisfireHandlingInstructionNextWithRemainingCount
    之前丢失的不管了，之后执行时间点和周期不会发生变化
    剩余次数=预定的总次数-丢失的次数

  4. WithMisfireHandlingInstructionNextWithExistingCount
    之前丢失的不管了，之后执行时间点和周期不会发生变化
    剩余次数=预定的总次数

  5. WithMisfireHandlingInstructionNowWithRemainingCount
    之前丢失的立即补执行一次，时间会从当前执行开始计算，时间周期不变
    剩余次数=预定的总次数-(丢失次数-1）

  6. WithMisfireHandlingInstructionNowWithExistingCount
    之前丢失的立即补执行一次，时间会从当前执行开始计算，时间周期不变
    剩余次数=预定的总次数-1

WithCalendarIntervalSchedule、WithDailyTimeIntervalSchedule、WithCronSchedule 有 3 种 Misfire 机制：

  1. WithMisfireHandlingInstructionDoNothing
    之前丢失的不管了，之后执行时间点和周期不会发生变化
    剩余次数=预定的总次数

  2. WithMisfireHandlingInstructionFireAndProceed
    之前丢失的立即补执行一次，之后执行时间点和周期不会发生变化
    剩余次数=预定的总次数-1

  3. WithMisfireHandlingInstructionIgnoreMisfires
    之前丢失的次数会马上全部补上，之后执行时间点和周期不会发生变化
    剩余次数=预定的总次数-丢失次数

注意：在测试过程中，一定要关注到默认超出60秒才会触发 Misfire，否则设置任何机制都无效。

ForJob

指定 Trigger 对应的 Job，对应的参数值可以是 jobName、 jobKey、jobDetail。但要注意，通过 ForJob 的方式设置的 Trigger ，当 Trigger 被加入调度器时，要确保对应的 Job 已经存在于调度器内。

Priority

当多个 Trigger 在同一个时间点触发一个 Job 的时候，可以通过设置 Trigger 优先级来控制执行顺序，Trigger 默认的优先级是 5，值越大优先级越高，最高10。

ModifiedByCalendar

如果希望在指定的周期内排除某些时间段不要执行 Job，我们需要在 Trigger 内增加 ModifiedByCalendar 配置，所有的 Calendar 既设置是排除，也可以是包含。Quartz 提供了以下几种 Calendar：

  1. DailyCalendar 排除一天中的某些时间段不执行

  2. WeeklyCalendar 排除星期中的一天或多天

  3. HolidayCalendar 排除特定的日期，精确到天

  4. MonthlyCalendar 排除月份中的某天，可选值为1-31，精确到天

  5. AnnualCalendar 排除每年中的某天，精确到天

  6. CronCalendar 使用表达式排除某些时间段不执行

先通过 AddCalendar 添加到调度器中，然后通过 ModifiedByCalendar 修改 Trigger，所以下面 Trigger 的配置在 11月11日是不会触发 Job 的。
